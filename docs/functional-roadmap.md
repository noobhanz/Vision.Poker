# Functional Product Roadmap

This roadmap focuses on turning Vision Poker from a strong prototype into a functional, testable product. The guiding principle is to build one reliable vertical slice first, then widen support.

## Phase 0: Product Guardrails

Goal: make sure the project has safe boundaries before scaling development.

- Complete the compliance review described in `docs/compliance-due-diligence.md`.
- Keep explicit non-goals in the README: no automated clicking, process injection, memory reading, poker-server communication, or hand-history scraping.
- Add a development/license bypass for local testing so backend setup does not block app validation.
- Decide the first supported environment: one poker client, one table layout, one operating system, one table size.

Exit criteria:

- Compliance questions are tracked.
- MVP target client and environment are chosen.
- Local app can run in dev mode without production licensing.

## Phase 1: Offline Vision Lab

Goal: prove parsing works without live capture.

- Add screenshot fixtures under `tests/fixtures/sample_frames/`.
- Store expected parsed state beside each image as JSON.
- Finish or extend `tools/replay_test.py` to run the full vision parser against fixtures.
- Add `tools/debug_frame.py` to output annotated screenshots showing ROIs, detected cards, OCR text, confidence, and parse status.
- Add fixture-based tests for ROI scaling, card parsing, OCR parsing, and full `GameState` parsing.

Exit criteria:

- One command reports field-level accuracy for cards, board, pot, bet, and street.
- Debug images clearly show what the parser saw.
- Failures can be diagnosed from saved annotated output.

## Phase 2: Card Recognition MVP

Goal: replace missing model assets with a reliable first recognizer.

- Start with fixed-slot card classification rather than YOLO object detection.
- Build rank and suit templates for known card ROIs.
- Normalize card crops: resize, grayscale, threshold, crop rank/suit corner, compare templates.
- Return card label and confidence from each slot.
- Keep the existing YOLO interface available for a future model-backed path.
- Add tests against real sample screenshots and synthetic crops.

Exit criteria:

- Hero and board cards are recognized from fixture screenshots.
- Template assets are versioned in the repo or generated by a documented tool.
- Missing/ambiguous cards return low confidence rather than false certainty.

## Phase 3: Poker-Specific OCR

Goal: make pot, bet, and stack reading stable enough for real use.

- Add per-skin OCR preprocessing: crop, upscale, grayscale, threshold, optional invert, denoise.
- Keep EasyOCR as the initial reader, but isolate it behind `OCREngine`.
- Add fallback parsing for common poker formats: `$1.25`, `1,250`, `50BB`, `2.5K`, all-in labels, and empty/no-bet states.
- Add confidence or parse status for OCR fields.
- Add fixture coverage for pot, bet-to-call, hero stack, and villain stacks.

Exit criteria:

- OCR succeeds on the selected MVP client screenshots.
- Invalid or missing text does not produce misleading metrics.
- Pot and bet errors are visible in debug output.

## Phase 4: Pipeline Reliability

Goal: make the live runtime path behave predictably.

- Use scaled ROIs for frame-buffer change detection.
- Avoid finding the poker window twice per frame; capture the exact rect already selected.
- Reset frame buffer when the active table/window changes.
- Add `street` and parse status to metrics or HUD state.
- Suppress recommendations or show a warning when confidence is below threshold.
- Add structured debug logs for capture failures, parse failures, OCR failures, and metric failures.

Exit criteria:

- Live capture uses the same coordinates as offline parsing.
- HUD does not display stale or overconfident recommendations.
- Moving/resizing/switching tables does not silently break parsing.

## Phase 5: Calibration Workflow

Goal: make unsupported layouts configurable by users or testers.

- Improve `tools/calibrate_roi.py` into a clear calibration workflow.
- Let users capture a table screenshot and mark hero cards, board cards, pot, bet, and stack regions.
- Save calibrated configs as `config/skins/custom_<name>.json`.
- Run an immediate parse test after calibration.
- Add support for relative coordinates when appropriate.

Exit criteria:

- A tester can calibrate a new skin without editing JSON by hand.
- The generated config can be replay-tested against the calibration screenshot.
- Calibration output includes enough metadata to reproduce the table layout.

## Phase 6: HUD and UX Polish

Goal: make the HUD useful and trustworthy.

- Display street, parse confidence, and low-confidence states correctly.
- Distinguish unavailable data from zero values.
- Add statuses such as `Waiting for table`, `No hero cards`, `Low confidence`, and `Ready`.
- Make recommendations conservative when inputs are incomplete.
- Add a settings surface for client, skin, HUD position, hotkey, opacity, and debug mode.

Exit criteria:

- HUD state matches parser state.
- Users can tell when the app is uncertain.
- Common settings are available without editing `.env`.

## Phase 7: Packaging and Installation

Goal: ship an app that contains the assets it needs.

- Decide packaging strategy:
  - lean template-based app,
  - heavy YOLO/EasyOCR bundle,
  - or hybrid asset download after install.
- Include required skin configs, templates/model assets, styles, and license dialogs in the build.
- Test the packaged app, not only source mode.
- Add macOS permissions guidance for Screen Recording and Accessibility.
- Add signing/notarization steps before public macOS distribution.

Exit criteria:

- Fresh install works on a clean machine.
- Packaged app can run the same demo/replay path.
- Required permissions and assets are present.

## Phase 8: Backend and Commercial Readiness

Goal: make licensing and payments safe enough for production.

- Add database migrations.
- Move secrets out of defaults and enforce required production config.
- Add webhook idempotency and better Stripe failure handling.
- Rate-limit signup, activation, validation, and checkout endpoints.
- Protect user/license lookup endpoints.
- Add operational logging and health checks.
- Add customer portal flow for subscription management.

Exit criteria:

- Backend can be deployed reproducibly.
- Payment and license state transitions are tested.
- Production configuration cannot accidentally use placeholder secrets.

## Phase 9: Wider Client Support

Goal: expand only after one client is reliable.

- Add additional sample fixtures per poker client.
- Build or calibrate skin configs per client/theme/table size.
- Track accuracy per client and per field.
- Add regression tests before claiming support.
- Consider YOLO/card-localization only when fixed-slot recognition is no longer sufficient.

Exit criteria:

- Each supported client has fixtures, expected JSON, ROI config, and passing replay tests.
- Support status is honest: experimental, beta, or stable.

## Recommended MVP Sequence

1. Choose one client and one table layout.
2. Add fixture screenshots and expected JSON.
3. Build debug-frame output.
4. Implement fixed-slot template card recognition.
5. Improve OCR preprocessing for that client.
6. Make replay tests pass.
7. Fix live pipeline reliability issues.
8. Run a local live demo.
9. Package a developer build.
10. Complete compliance review before public or paid release.
